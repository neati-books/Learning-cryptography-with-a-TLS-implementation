Introduction

라이브러리는 사용되어야 할 것이지, 배워야 할 것이 아니다.

세상에는 배워야 하고 외워야 하는 것들이 많습니다. 우리는 여기에 하나 더 더하고 싶지 않습니다. eZ 프레임워크는 라이브러리는 사용되어야 할 것이지, 배워야 할 것이 아니다라는 모토를 기반으로 만들어졌습니다. eZ 프레임워크는 매우 쉬운 서버측 웹개발 프레임워크입니다. eZ 프레임워크로 개발하기 위해 외워야 할 것은 몇 가지 밖에 안됩니다. 나머지는 프로그래머의 응용력에 달려 있습니다. eZ 프레임워크는 단순한 웹 개발 환경을 제공합니다. 프로그래머는 한 명의 접속자와 연속적인 메세지 교환을 한다고 생각하고 웹 앱을 개발하면 됩니다. 웹사이트를 하나의 클래스로 정의하기에 구조적인 프로그래밍을 하기에도 용이합니다.

Liabrary is to be used, not to be learned.

There are so many things to learn and memorize. We don't want to add one more burden. eZ Framework is a very easy server side web development framework. You just need to memorize just a few things to develop with eZ framework. The rest depends on the developer's adaptability. eZ framework provides very simple web development environment. Developer can just think that the program is exchanging messages with one connector. Web site is defined by a class so that structured programming is very convinient.

Stateful한 가상 프로그래밍 환경 제공

C++은 성숙하고 다양한 기능을 제공하며, 빠른 언어임에도 불구하고, 웹 개발에 있어서는 거의 사용되지 않았습니다. 그 가장 큰 이유는 웹 개발은 일반적인 소프트웨어 개발과는 다르게, 웹에 접근하는 사용자가 보는 페이지 별로 분절되어 있기 때문입니다. 그렇기 때문에 본격적인 언어보다는 스크립트형 언어가 웹 개발에 더욱 많이 쓰이고 있습니다. 또, 웹 개발에서는 로그인 등의 상태 정보를 저장하기 위해서 쿠키를 사용하는 등의 복잡한 절차를 이용해야 합니다. eZ 프레임워크는 하나의 브라우저가 연속적으로 연결되어 있는 것과 같은 Stateful한 가상적인 환경을 제공합니다. 이 가상적인 환경이 페이지간의 분절을 없애줍니다. 이 덕분에 프로그래머는 편하게 C++의 변수에 여러 페이지에 걸쳐 기억해야 할 정보를 할당할 수 있습니다. 더 이상 로그인 되어 있는 상태인지 확인하기 위해 쿼리를 날린다거나, 쿠키를 확인하는 복잡한 작업을 할 필요가 없습니다. 그저 멤버 변수나 지역 변수를 확인하면 됩니다. 홈페이지의 영국 국기를 클릭하셨다면 이 페이지가 영어로 보일 것입니다. 언어를 어느 것으로 선택했는지를 eZ 프레임워크는 그냥 멤버 변수에 저장해 기억할 수 있고, 이를 여러 페이지에 걸쳐 사용할 수 있습니다.

Virtual programming environment with State

C++ is a mature, versatile and very fast language. But it was almost never used in web development. The biggest reason is that web development is different from normal software development. It is segmented by pages users are seeing. This is why script like languages are prefered, rather than full edged languages. And, we need to use cookie or some other complex methods to store informations like login status. eZ framework provide a stateful virtual environment that a browser is constantantly connected to a web server. This virtual environment removes the segmentation of web apps. Thanks to this a programmer can just store those informations that should be sustained through multiple pages to a normal C++ variables. This is very convinient. No more queries or cookies to verify the user is logged in. Just look at the local variables. If you clicked Korean flag at the home page, this page will be seen as Korean. The language you selected can be stored in a member variable, this information can be used across many pages.

미들 서버

eZ 프레임워크는 자신의 웹서버로 돌릴 실행파일을 직접 만듭니다. 아파치나 톰캣 같은 덩치 큰 서버 프로그램을 돌릴 필요가 없습니다. eZ 프레임워크는 https 암호화와 가상 환경을 제공하는 미들 서버와 웹 서버로 나뉘어 있습니다. 미들서버는 프레임워크에서 곧장 사용할 수 있는 기성품의 형태로 제공합니다. 개발자는 웹 서버만을 개발하면 됩니다. 이 미들서버를 사용하면 복잡하게 설정을 할 필요가 없이 쉽게 암호화 된 웹사이트를 구현할 수 있습니다. 이 미들서버는 만약 아파치 서버와 외부 브라우저 사이에 위치시킨다면 아파치 서버의 설정에 아무런 변경 없이 암호화를 제공할 수 있을 정도로 유연합니다. 다음에 직접 접속해 보세요. 미들 서버로 Apache Tomcat에 암호화 접속 제공. 물론, eZ 프레임워크로 개발한 웹 서버와 브라우저 사이에 위치시킨다면, Stateful한 편리한 개발 환경까지 제공합니다.

Middle Server

eZ framework directly makes a executable to run as a web server. We do not need to run behemothian Apache or tomcat server. eZ framework is composed of middle server and web server. Middle server enables https encryption and distributing connections to web servers. This middle server is very flexible that it can just provide a https connection to apache server without any configuration. Try this link to see how simple it is. Middle server is providing Encrypted Connection to Apache Tomcat. Of course, if it is located between web server made with eZ framework and browser, it can provide a stateful virtual environmet.

쉬운 템플릿 사용

eZ 프레임워크는 HTML, CSS, Java Script 등을 템플릿으로 사용할 수 있으며, 복잡한 템플릿 문법이 없습니다. 단지 swap, append 등의 함수를 제공합니다. 정규표현식에 친숙하다면 이를 사용해서 문자열을 조작할 수도 있습니다. 템플릿은 정적인 웹페이지와 아무 차이가 없습니다. 정적인 웹페이지를 템플릿으로 만들어 두고, 서버측에서 이 페이지를 필요에 따라 변환하여 브라우저에 보낸다고 생각하면 쉽습니다.

Easy Template Usage

eZ framework can use HTML, CSS, Java Script as template. But eZ framework does not have complex template grammar. We just provide swap, append functions. If you are familiar to regular expressions, you can use it. Template is not different from static web page. You can think template usage like this.

  • Make a static web page
  • Transform it in server side if needed
  • Send it to browser

MySql 편의 클래스 라이브러리와 다양한 라이브러리의 사용 가능성

eZ 프레임워크는 MySql 데이터 베이스에 데이터를 입력하고 출력하는 편의 클래스를 제공하고 있습니다. 그 외의 데이터 베이스의 경우에는 개발자가 스스로 라이브러리를 사용하여 링크하면 됩니다. C/C++ 언어는 오랜 기간 누적되어온 다양한 라이브러리가 있습니다. 이를 모두 간단히 링크하여 사용할 수 있다는 것은 C++언어를 사용하는 잇점입니다. 물론, python이나 다른 언어를 사용하여 접근할 수도 있습니다. 웹 서버 프로그램이 다른 실행 파일이나 스크립트 파일을 실행시키거나, pybind 등을 이용하여 파이선 함수를 부르는 것은 그다지 어렵지 않습니다.

MySql Utility class and easy adoptation of various libraries

eZ framework has a utility class for MySql database. For other databases, developers can use appropriate library or use python or other language module. C/C++ language has long accumulated diverse libraries. It is a privilege to use these with simple linking. Of course, Web server program can easily run script file, and call python functions using pybind etc.

기본 메커니즘

eZ 프레임워크는 "웹서버는 문자열을 받아서 처리한 후, 문자열을 돌려주는 것이다"라는 매우 단순한 통찰을 기반으로 만들어졌습니다. eZ 프레임워크는 문자열을 받아서 그 요청이 GET이던 POST이던 multi-form file 업로드이던 ajax 이건 상관없이 변수이름과 그 값을 nameNvalue_라는 일종의 map에 저장합니다. 요청된 페이지는 requested_document_라는 변수에 저장합니다. 그리고, 요청된 페이지의 템플릿이 있을 경우에는 그 내용을 content_에 저장합니다. 그리고, 이 content_값은 브라우저에게 보내질 것입니다. 그러나, 브라우저에게 이 값을 보내기 전에 서버단에서의 처리를 위하여, process함수를 부릅니다. 이 함수가 바로 서버측 개발자가 구현해야 할 함수입니다. 이 process 함수 안에서 제공된 nameNvalue_ 변수를 이용하여, requested_document_에 따라 이미 템플릿 값으로 생성되어 있는 content_를 변형시키는 것이 목적입니다. 만약 서버측에서 아무런 해야 할 일이 없다면 아무 것도 하지 않아도 됩니다. 튜토리얼 1에서 이러한 정적인 웹페이지의 예를 볼 수 있습니다.

Basic mechanism

eZ framework is based on the simple insight that a web server just get string and process it and return string. eZ framework stores all the parameters into nameNvalue_(map) regardless of GET, POST, multi-form file. It stores the requeseted document to variable requeseted_document_. If the template of requeseted document is present, it will be copied to content_ variable. This content_ will be sent back to browser. But before this, eZ framework will call a function to process server side procedures. This function(named process) is what developers will implement. A developer should transform content_ variable formed by the template in this process function, using nameNvalue_ variable. If there is nothing to be done in server side, no programming is needed. Tutorial 1 is such example.

어떻게 작동하는가?

eZ 프레임워크는 미들 서버와 웹 서버로 나뉘어 집니다. 미들 서버는 웹 브라우저와 웹 서버 사이에 위치하여, 암호화된 연결과 가상적인 연속적으로 연결된 환경을 제공합니다. 미들 서버는 외부에서 온 패킷을 브라우저별로 분류하여, 자신의 내부에 일종의 대리 클라이언트를 만듭니다. 이 클라이언트는 웹 서버와 지속적인 접속을 유지함으로써, 가상적인 연속 연결 상태를 만듭니다. 패킷을 브라우저별로 분류하기 위해, 세 가지의 방법을 미들 서버는 사용합니다.

  • TLS 1.3 버전의 세션 재개 티켓을 이용
  • TLS 1.2 버전의 세션 아이디를 이용
  • 쿠키를 이용

How does it work?

eZ framework is made up of middle server and web server. Middle server is located between web browser and web server. It provide TLS encryption and virtual stateful connection. Middle server categorize packets by browsers. Middle server makes some proxy client inside of itself to connect to web server continuously. Middle server uses three methods to identify the packets according to browser.

  • TLS 1.3 session resumption ticket
  • TLS 1.2 session ID
  • Cookie

위의 그림에서는 로컬 머신에 웹 서버가 있을 경우만 표현하고 있지만, 실제적으로는 웹 서버는 다른 지역 네트워크에 있어도 상관이 없습니다. 미들 서버는 여러 개의 머신에 있는 웹 서버에 접속을 분배할 수 있습니다.
